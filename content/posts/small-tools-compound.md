A five-line script can look trivial in isolation. Used every day, it becomes
a piece of infrastructure.

Small tools are powerful because they fit the shape of the problem closely.
They do not need a roadmap, an onboarding flow, or a universal abstraction.
They need to remove one recurring piece of friction reliably.

## Look for repeated decisions

Repetition is not only repeated typing. It is also repeated judgment:

- Which command should I run?
- What belongs in this document?
- Did I forget a release step?
- How should this file be named?

A shell alias handles the first. A template handles the second. A checklist
handles the third. A naming function handles the fourth.

## Keep the tool close to the work

The shorter the distance between a tool and its context, the easier it is to
maintain. Project-specific scripts should live with the project. Writing
templates should live beside the writing. A short README should explain the
non-obvious parts.

## Let usefulness earn complexity

Start with the smallest version that removes the friction. If it keeps being
used, improve its error messages. If other people adopt it, document its
interface. If its inputs become varied, then consider configuration.

Complexity should be paid for by demonstrated usefulness—not anticipated
importance.
